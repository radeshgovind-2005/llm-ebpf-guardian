# Code of Conduct

## Our Commitment

**llm-ebpf-guardian** is a technically demanding, security-focused project.
We are committed to keeping it a place where contributors of all experience
levels — from kernel hackers to ML engineers — can participate without
harassment or gatekeeping.

We pledge to make participation in this project a respectful experience for
everyone, regardless of age, background, disability, ethnicity, gender identity,
level of experience, nationality, personal appearance, race, religion, or
sexual identity and orientation.

---

## Our Standards

### Expected behaviour

- **Be precise and constructive** in code review — critique the code, not the author
- **Assume good intent** — most mistakes are honest ones, especially in a complex kernel/userspace project
- **Be patient with newcomers** — eBPF, BPF CO-RE, and syscall tracing have steep learning curves
- **Credit others** — if you build on someone's idea or fix, say so
- **Keep security discussions private** — follow the [Security Policy](SECURITY.md) for vulnerability reports; do not post exploit details publicly before a patch is released

### Unacceptable behaviour

- Harassment, insults, or personal attacks in issues, PRs, or discussions
- Dismissing contributions because of who wrote them rather than what they contain
- Publishing others' private information without consent
- Using the project's security research context as cover for actual malicious activity
- Sustained disruption of technical discussions

---

## Scope

This Code of Conduct applies to all project spaces: GitHub Issues, Pull
Requests, Discussions, the repository wiki, and any other communication
channel where someone is representing this project.

It also applies outside project spaces when someone's conduct there could
reasonably reflect on the project or its community.

---

## Enforcement

### Reporting

If you experience or witness unacceptable behaviour, report it privately:

1. Use GitHub's **"Report" button** on the specific comment or profile, **or**
2. Contact the maintainer directly via the email on the [GitHub profile](https://github.com/radeshgovind-2005)

Reports are treated confidentially. Do not open a public Issue to report
conduct — this exposes the reporter and can escalate the situation.

### What happens after a report

| Step | Timeline |
| ---- | -------- |
| Acknowledgement of report | Within **48 hours** |
| Investigation complete | Within **7 days** |
| Decision communicated to reporter | Within **10 days** |

### Consequences

Maintainers will use the following graduated response:

1. **Correction** — private message explaining what was inappropriate and why
2. **Warning** — formal warning with specific behavioural requirements; further violations escalate
3. **Temporary ban** — temporary removal from all project interaction (length depends on severity)
4. **Permanent ban** — permanent removal for sustained, severe, or bad-faith violations

Decisions are made by the maintainer. Appeals can be sent to the same
contact address with "Appeal:" in the subject.

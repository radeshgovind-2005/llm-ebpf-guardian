## 1. What this document is

An **Architectural Decision Record (ADR)** is a document that captures an important technical decision made during a project — including the context that led to it, the options that were considered, and the reasoning behind the final choice.

This ADR covers the decisions made during the initial setup of `llm-ebpf-guardian` and the completion of Phase 1. It is written to be understandable by both non-technical readers and experienced engineers.

---

## 2. The problem this project solves

### The risk in agentic AI

Modern AI systems — called **agents** — do not just answer questions. They write code and execute it. An agent given the task "write a config file" will generate a Python script and run it on a real computer.

This creates a security problem: the code the agent generates might do something the operator never intended. A script asked to write a file might also open a network connection and send data to a remote server. The model does not "know" it is doing something wrong. It simply generates code that runs.

This is not a hypothetical risk. It is the core operational risk in any system that runs LLM-generated code.

### What Guardian does

`llm-ebpf-guardian` enforces a **runtime security boundary** between what an agent was asked to do and what its process is actually doing. It works at the operating system level — below the application, below the Python runtime — by intercepting every system call the process makes.

The central idea:

> A process asked to write a file should call `openat` and `write`.
> If it calls `socket` and `connect`, something is wrong.
> Guardian detects that mismatch and kills the process before damage occurs.

## 3. Background: key concepts explained

### What is a system call (syscall)?

Every program that runs on a computer eventually needs to interact with the hardware — to read a file, open a network connection, or create a new process. It cannot do this directly. Instead, it asks the operating system kernel to do it on its behalf. These requests are called **system calls**, or syscalls.

Examples:
- `openat` — open a file
- `write` — write data to a file or socket
- `socket` — create a network socket
- `connect` — connect to a remote server
- `execve` — start a new process
- `ptrace` — attach to and control another process (used by debuggers)
- `mount` — attach a filesystem

Every action a process takes ultimately maps to one or more syscalls. Monitoring syscalls gives you a complete, unforgeable record of what a process is doing.

### What is eBPF?

**eBPF** (Extended Berkeley Packet Filter) is a technology built into the Linux kernel that lets you run small, sandboxed programs inside the kernel itself — without modifying the kernel's source code or loading a kernel module.

An eBPF program can attach to a syscall and run every time that syscall is invoked by any process. It can inspect the arguments, record the event, and pass information to userspace — all with near-zero performance overhead because it runs directly in the kernel.

Think of it as placing a silent observer at every door in the operating system. Every time a process walks through a door, the observer notes who it is, where they are going, and when.

### What is BTF?

**BTF** (BPF Type Format) is metadata embedded in the Linux kernel that describes all of the kernel's internal data structures. It allows eBPF programs to be written in a portable way — they can access kernel data structures by name, and the kernel figures out the correct memory layout at runtime.

For eBPF programs to work, the kernel must have been compiled with BTF support enabled. This manifests as a file at `/sys/kernel/btf/vmlinux` on the host system. If that file exists, the kernel supports modern eBPF development.

### What is libbpf?

**libbpf** is the standard C library for loading and attaching eBPF programs. It reads compiled eBPF bytecode, handles the BTF-based portability logic (called CO-RE: Compile Once, Run Everywhere), and attaches programs to kernel tracepoints.

### What is bpftool?

**bpftool** is a command-line utility that communicates directly with the kernel's BPF subsystem. Among other things, it can extract the kernel's BTF data and format it as a C header file (`vmlinux.h`). This header file is what eBPF programs include to access kernel data structures.

## 4. Decision: development environment

### Context

The project targets Linux kernel features (eBPF, BTF, syscall tracepoints). Development is being done on a MacBook Pro (Apple Silicon, arm64). MacOS does not have a Linux kernel and cannot run eBPF programs natively.

Two approaches were considered:

| Option | Description | Tradeoff |
|---|---|---|
| **Docker Desktop** | Industry-standard container runtime for Mac | Uses a lightweight Linux VM (linuxkit) that may lack proper BTF support |
| **OrbStack** | Lightweight alternative container runtime for Mac | Runs a full Linux kernel with BTF enabled; faster and lower memory usage |

### What happened

The project initially used Docker Desktop. The container environment built successfully, but when attempting to use `bpftool` to extract the kernel's BTF data, the operation failed:

```
Error: failed to load BTF from /sys/kernel/btf/vmlinux: Invalid argument
```

The root cause: Docker Desktop's linuxkit kernel (version 6.12) was too new for the system-installed `bpftool` (version 5.15), which could not parse the newer BTF format. The version gap was too large to bridge with the packages available on Ubuntu 22.04.

### Decision

**Switch to OrbStack.**

OrbStack provides a full Linux kernel environment (version 6.17 in this case) with proper BTF support, and integrates seamlessly with VS Code Dev Containers. The migration required:

1. Installing OrbStack via Homebrew: `brew install orbstack`
2. Switching the active Docker context: `docker context use orbstack`
3. Rebuilding the dev container under the new runtime

The VS Code `devcontainer.json` configuration required no changes. OrbStack is a drop-in replacement for Docker Desktop at the container API level.

**This decision is permanent for this project.** Any contributor developing on macOS should use OrbStack.

## 5. Decision: toolchain for eBPF compilation

### Context

eBPF programs are written in a restricted subset of C and compiled to BPF bytecode using **Clang**, not the standard GCC compiler. The compiled bytecode is then loaded into the kernel at runtime.

The dev container needs:
- `clang` and `llvm` — to compile eBPF C code to BPF bytecode
- `libbpf-dev` — the C library headers for writing portable eBPF programs
- `linux-headers-generic` — kernel headers for type definitions
- Build utilities: `build-essential`, `pkg-config`, `libelf-dev`, `zlib1g-dev`

### What was attempted

The initial `Dockerfile.dev` included `bpftool` as a standalone package. On Ubuntu 22.04 arm64, this package does not exist — `bpftool` is instead distributed as part of `linux-tools-generic`, a meta-package that installs tools matching the currently running kernel version.

However, the system-installed `bpftool` (version 5.15, tied to Ubuntu 22.04's LTS kernel) was incompatible with OrbStack's kernel (version 6.17), causing the BTF extraction to fail.

### Decision

**Build bpftool from source** using the [libbpf/bpftool](https://github.com/libbpf/bpftool) repository.

Building from source produces the latest version of bpftool, which supports any kernel version. It is compiled with the same libraries available in the container, eliminating the version mismatch problem entirely.

The build requires these additional packages: `libcap-dev`, `binutils-dev`, `libssl-dev`.

The final `Dockerfile.dev`:

```dockerfile
FROM ubuntu:22.04

ARG DEBIAN_FRONTEND=noninteractive

RUN apt-get update && apt-get install -y \
    clang llvm libbpf-dev linux-headers-generic \
    linux-tools-generic linux-tools-common \
    python3 python3-pip python3-venv \
    curl git build-essential pkg-config \
    libelf-dev libcap-dev zlib1g-dev binutils-dev libssl-dev \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /app
COPY requirements.txt .
RUN pip3 install --no-cache-dir -r requirements.txt

# Build bpftool from source to match any kernel version
RUN git clone --depth 1 https://github.com/libbpf/bpftool.git /tmp/bpftool \
    && cd /tmp/bpftool && git submodule update --init \
    && cd src && make && make install \
    && rm -rf /tmp/bpftool

CMD ["/bin/bash"]
```

The resulting binary is `bpftool v7.7.0`, which successfully reads BTF data from OrbStack's kernel 6.17.

## 6. Decision: bpftool installation strategy

### Context

`vmlinux.h` — the kernel type header that eBPF programs include — is generated by running:

```bash
bpftool btf dump file /sys/kernel/btf/vmlinux format c > kernel/vmlinux.h
```

This file is 189,298 lines long and contains every data structure defined in the Linux kernel. It is specific to the running kernel version and should not be committed to version control as a static file, because it will be different on every kernel.

### Decision

`vmlinux.h` is **generated at build time** inside the dev container, not committed to the repository. The `Makefile` provides a `make vmlinux` target that runs the bpftool command. Engineers setting up the project for the first time run `make vmlinux` as the first build step.

The `Makefile` uses the full path to bpftool to avoid relying on `$PATH` resolution:

```makefile
BPFTOOL := $(shell which bpftool || echo /usr/sbin/bpftool)
```

`kernel/vmlinux.h` is listed in `.gitignore`.

## 7. Decision: Ollama for anomaly scoring

### Context

The anomaly scorer (Phase 3) needs to interpret a natural language task prompt — for example, "write a JSON config file to /tmp/output.json" — and derive from it the set of syscalls that process is legitimately expected to make.

This is a language understanding problem. Three options were considered:

| Option | Description | Tradeoff |
|---|---|---|
| **Rule-based matching** | Keyword search over the prompt | Fast, but brittle. "Write a report" and "exfiltrate data" could match the same keywords. |
| **External LLM API** | Call OpenAI or Anthropic's API | Accurate, but requires internet access, incurs cost, and sends potentially sensitive task prompts to a third party — which is ironic for a security tool. |
| **Local model via Ollama** | Run a model locally on the developer's Mac | No cost, no data leaves the machine, fits the security ethos of the project. |

### Decision

**Use Ollama with a local model.**

Ollama runs on the host Mac and exposes a REST API at `localhost:11434`. The dev container forwards this port, so Python code inside the container can reach it via `http://host.docker.internal:11434`.

The scorer sends the task prompt to the local model and asks it to return a JSON object listing expected syscalls and allowed file path prefixes. This gives the scoring system genuine language understanding without any external dependencies.

The default model is `llama3.2`, but any model available in Ollama can be substituted by changing one constant in `scorer.py`.

## 8. What was built in Phase 1

Phase 1 establishes the kernel layer: the eBPF probe that intercepts syscalls and emits structured events.

### Files created

**`kernel/guardian.h`** — shared data structures used by both the kernel eBPF program and the userspace Python loader. Defines the `guardian_event` struct (what each intercepted syscall looks like) and the `syscall_id` enum (which syscalls are monitored).

**`kernel/guardian.bpf.c`** — the eBPF probe. Written in restricted C, compiled to BPF bytecode. Attaches to six kernel tracepoints:

| Tracepoint | Syscall | Why it matters |
|---|---|---|
| `sys_enter_execve` | `execve` | Detects child process spawning |
| `sys_enter_socket` | `socket` | Detects network socket creation |
| `sys_enter_connect` | `connect` | Detects outbound network connections |
| `sys_enter_openat` | `openat` | Detects file access |
| `sys_enter_ptrace` | `ptrace` | Detects process injection attempts |
| `sys_enter_mount` | `mount` | Detects filesystem manipulation |

For each intercepted syscall, the probe records: timestamp, process ID, process name, and syscall-specific detail (file path, socket family, remote address). It writes this as a `guardian_event` to a BPF ring buffer that userspace reads continuously.

A BPF hash map called `target_pid` allows userspace to configure which process to monitor. If set to 0, all processes are monitored (used for testing).

**`userspace/loader.py`** — the Python process that loads the compiled eBPF object, attaches the tracepoints, and reads events from the ring buffer. Phase 1 includes the event struct definition (via `ctypes`) and a demonstration mode that prints colour-coded synthetic events without requiring the probe to be running. Full ring buffer integration is Phase 2.

**`userspace/scorer.py`** — the anomaly scoring engine. Takes a task prompt, queries Ollama to build a syscall profile, and scores each incoming event against that profile. Phase 1 includes the complete scoring logic; it is integrated with the live event stream in Phase 3.

**`profiles/default_policy.yaml`** — a human-readable configuration file defining syscall allowlists for common task types (file write, file read, data processing, network fetch). Used as a fallback when Ollama is unavailable.

**`Makefile`** — two primary targets:
- `make vmlinux` — generates `kernel/vmlinux.h` from the running kernel's BTF data
- `make probe` — compiles `guardian.bpf.c` to `guardian.bpf.o` using Clang

### Verified outcomes

At the end of Phase 1, the following has been confirmed working inside the dev container:

```
$ bpftool btf dump file /sys/kernel/btf/vmlinux format c > kernel/vmlinux.h
$ wc -l kernel/vmlinux.h
189298 kernel/vmlinux.h

$ make probe
Compiling eBPF probe...
Done: kernel/guardian.bpf.o

$ python3 userspace/loader.py --task "write a JSON config file"
llm-ebpf-guardian
Task: write a JSON config file
openat   pid=12345 comm=python3         path=/tmp/output.json
socket   pid=12345 comm=python3         path=-
connect  pid=12345 comm=python3         path=-
ptrace   pid=12345 comm=python3         path=-
```

## 9. Known constraints and future work

### Constraint: arm64 / macOS development environment

This project is developed on Apple Silicon (arm64). The eBPF probe compiles and the BTF dump works correctly under OrbStack. However, production eBPF deployment targets x86_64 Linux servers. The `Makefile` detects the host architecture automatically and sets `__TARGET_ARCH_arm64` or `__TARGET_ARCH_x86` accordingly. This has not yet been tested on x86_64.

### Constraint: `vmlinux.h` is kernel-specific

The generated `vmlinux.h` is valid only for the kernel version it was extracted from. If the dev container is rebuilt with a different kernel (e.g. after an OrbStack update), `make vmlinux` must be re-run. This is expected behaviour and is documented in the project README.

### Constraint: libbpf-python not yet integrated

The Python loader (`loader.py`) currently operates in a demonstration mode with synthetic events. Real integration with the BPF ring buffer requires `libbpf-python`, which has no pre-built arm64 package on Ubuntu 22.04. Phase 2 will resolve this by building it from source or using `ctypes` to call the installed libbpf directly.

### Upcoming phases

| Phase | Branch | Goal |
|---|---|---|
| Phase 2 | `phase/2-userspace-loader` | Real ring buffer integration; `loader.py` reads live syscall events from the running probe |
| Phase 3 | `phase/3-anomaly-scorer` | Integrate `scorer.py` with the live event stream; Ollama profiles task prompts in real time |
| Phase 4 | `phase/4-policy-engine` | `policy.py` issues `SIGKILL` when the anomaly threshold is breached |
| Phase 5 | `phase/5-demo-polish` | `docker-compose.yml` demo; guardian container monitors a target container running LLM-generated code |
